### Prompt 1.2 — Auth module

```
In /src/auth/zohoAuth.ts, implement a Zoho OAuth2 token manager with:

1. A ZohoAuthManager class that:
   - Stores the refresh token, client ID, client secret, and domain from env
   - Has a getAccessToken() method that:
     * Checks if current access token is valid (not expired, with 60s buffer)
     * If expired or missing, calls POST https://{ZOHO_DOMAIN}/oauth/v2/token with grant_type=refresh_token
     * Caches the new token with its expiry timestamp
     * Returns the valid access token
   - Has a refreshAccessToken() private method
   - Retries up to 3 times on network failure with exponential backoff

2. Export a singleton instance

3. Handle these edge cases:
   - Zoho returns 401 mid-session (token invalidated externally) → force refresh
   - Refresh token itself is expired → throw a descriptive AuthError with remediation message
   - Network timeout → retry with backoff, then throw

4. Write a unit test in /tests/auth.test.ts that mocks axios and tests: valid cache hit, expired token refresh, retry on failure.
```

## 📋 Phase 2 — Zoho API Client

### Prompt 2.1 — Core API client with pagination

```
In /src/api/zohoClient.ts, build a robust Zoho Recruit API client class ZohoRecruitClient with these methods:

1. searchCandidates(filters: SearchFilters): Promise<Candidate[]>
   - POST to /recruit/v2/Candidates/search
   - Accepts filters: { skills?: string[], experience_min?: number, experience_max?: number, keywords?: string[], location?: string }
   - Handles pagination automatically: loop through pages (200 records/page max) until all results fetched or a cap of 1000 records
   - Returns normalized Candidate[] array

2. getCandidateDetails(candidateId: string): Promise<CandidateDetail>
   - GET /recruit/v2/Candidates/{id}
   - Returns full profile including skills, experience, current_employer, email, mobile

3. searchByKeyword(query: string): Promise<Candidate[]>
   - GET /recruit/v2/Candidates/search?criteria=...
   - Builds Zoho COQL-style criteria string from the query

4. Define TypeScript interfaces:
   - SearchFilters, Candidate, CandidateDetail in /src/api/types.ts

5. All methods must:
   - Auto-inject Bearer token via ZohoAuthManager
   - Retry 3x on 429 (rate limit) with Retry-After header respect
   - Retry 2x on 5xx with exponential backoff
   - Log each request/response summary via winston (no PII in logs at INFO level, full detail at DEBUG)
   - Throw typed errors: ZohoApiError, RateLimitError, NotFoundError

6. Write integration test stubs in /tests/zohoClient.test.ts using msw (mock service worker) to mock the Zoho endpoints.
```

## 📋 Phase 3 — LLM Integration

### Prompt 3.1 — JD Parser using Claude

```
In /src/llm/jdParser.ts, implement a JDParser class that uses the Anthropic Messages API (claude-sonnet-4-20250514) to extract structured data from a raw Job Description string.

The parsed output must conform to this Zod schema (define in /src/llm/schemas.ts):
{
  role_title: string,
  required_skills: string[],       // e.g. ["Java", "Spring Boot", "Microservices"]
  preferred_skills: string[],
  min_experience_years: number,
  max_experience_years: number,
  education_requirements: string[],
  domain_keywords: string[],       // e.g. ["banking", "BFSI", "capital markets"]
  seniority_level: "junior" | "mid" | "senior" | "lead" | "architect",
  location_preference: string | null,
  must_have: string[],             // non-negotiable requirements
  nice_to_have: string[]
}

Implementation requirements:
1. Read the system prompt from /prompts/jd_parser_system.txt
2. Send the raw JD as user message
3. Instruct the model to respond ONLY in valid JSON matching the schema above
4. Parse and validate response with Zod — if validation fails, retry once with an error-correction prompt
5. Handle: empty JD, JD in non-English, extremely long JD (>8000 tokens — chunk it)
6. Log token usage per call

Also create /prompts/jd_parser_system.txt with an expert-crafted system prompt. The prompt must:
- Instruct the model to act as a senior technical recruiter and skills taxonomy expert
- Emphasize extracting implicit skills (e.g. "financial domain experience" implies knowledge of FIX protocol, trade lifecycle)
- Distinguish must-have vs nice-to-have from JD language cues ("must", "required", "preferred", "plus")
- Output ONLY raw JSON, no markdown fences, no explanation
```

### Prompt 3.2 — Search Query Generator

```
In /src/llm/queryGenerator.ts, implement a SearchQueryGenerator class that takes the parsed JD (ParsedJD type) and generates optimized Zoho Recruit search parameters.

Method: generateSearchStrategies(parsedJD: ParsedJD): Promise<SearchStrategy[]>

It must return an array of 3 search strategies ordered by precision → recall:
1. PRECISION: Top 5 must-have skills + exact experience range
2. BALANCED: Top 8 skills (required + preferred) + relaxed experience (±2 years)  
3. RECALL: Domain keywords + seniority level + broad skill terms

Each SearchStrategy has:
{
  strategy_name: "precision" | "balanced" | "recall",
  filters: SearchFilters,
  zoho_criteria_string: string,   // Zoho COQL format: ((Skill_Set:contains:Java) AND (Experience:>=:5))
  rationale: string
}

Use Claude to generate the COQL criteria strings. Read system prompt from /prompts/query_generator_system.txt.

Create /prompts/query_generator_system.txt that instructs Claude to:
- Know Zoho Recruit COQL syntax: field:operator:value, AND/OR, parentheses grouping
- Valid operators: equals, contains, starts_with, between, >=, <=
- Key Zoho Candidate fields: Skill_Set, Experience_in_Months, Current_Location, Highest_Qualification, Candidate_Status
- Generate syntactically valid COQL — no hallucinated field names
- Include a self-check step before outputting
- Output only valid JSON array
```

### Prompt 3.3 — Candidate Ranker

```
In /src/llm/candidateRanker.ts, implement a CandidateRanker class.

Method: rankCandidates(candidates: CandidateDetail[], parsedJD: ParsedJD): Promise<RankedCandidate[]>

Algorithm (hybrid — do NOT use LLM for every candidate, it's too expensive):

Step 1 — Keyword Scoring (no LLM, fast):
For each candidate compute:
- skill_match_score: (matched required skills / total required skills) * 60
- experience_score: 20 if in range, 10 if ±2 years, 0 if far off
- domain_score: (matched domain keywords / total domain keywords) * 20
- keyword_total = skill_match_score + experience_score + domain_score

Step 2 — LLM Deep Scoring (only for top 20 by keyword_total):
Send batches of 5 candidates to Claude with the JD. Claude scores each 0-100 and provides:
- match_percentage: number
- matched_skills: string[]
- missing_critical_skills: string[]
- fit_summary: string (2 sentences max)
- recommendation: "strong_fit" | "possible_fit" | "weak_fit"

Step 3 — Final Score:
final_score = (keyword_total * 0.4) + (llm_score * 0.6)

Return RankedCandidate[] sorted by final_score descending.

Read LLM system prompt from /prompts/ranker_system.txt.

Create /prompts/ranker_system.txt that:
- Instructs Claude to act as an unbiased technical screening expert
- Score based ONLY on technical fit, not name/gender/age signals
- Be calibrated: 90%+ = exceptional match, 70-89% = strong, 50-69% = possible, <50 = weak
- Return only JSON array, strictly no explanation outside JSON
- Include confidence field if evidence for a skill is ambiguous
```

## 📋 Phase 4 — Pipeline Orchestration

### Prompt 4.1 — Main pipeline

```
In /src/pipeline/sourcingPipeline.ts, implement the SourcingPipeline class that orchestrates the full end-to-end flow:

async run(jdText: string): Promise<SourcingResult>

Flow:
1. Parse JD → JDParser.parse(jdText) → ParsedJD
2. Generate search strategies → SearchQueryGenerator.generateSearchStrategies(parsedJD) → SearchStrategy[]
3. Execute searches in sequence (precision first):
   a. Run precision search → collect candidates
   b. If results < 10, run balanced search → merge + deduplicate by candidateId
   c. If results still < 10, run recall search → merge + deduplicate
   d. Cap total unique candidates at 100 before ranking
4. Fetch full details for each candidate (with concurrency limit of 5 parallel requests)
5. Rank candidates → CandidateRanker.rankCandidates()
6. Return SourcingResult

SourcingResult type:
{
  job_title: string,
  total_searched: number,
  total_ranked: number,
  search_strategies_used: string[],
  candidates: RankedCandidate[],
  execution_time_ms: number,
  token_usage: { input: number, output: number, cost_usd: number }
}

Edge cases to handle:
- Zero results from all 3 strategies → return empty list with a suggestions field explaining what search terms were tried
- Zoho API down → fail fast with clear error, do not call LLM
- JD too vague (parsed JD has < 3 required skills) → prompt user to provide more detail, still proceed with what's available
- Duplicate candidates across strategies → deduplicate by email OR phone (not just ID)
- Concurrency: use p-limit to cap parallel Zoho detail fetches at 5
```

### Prompt 5.1 — Output formatter

```
In /src/output/formatter.ts, implement three output formatters for SourcingResult:

1. formatTable(result: SourcingResult): string
   - ASCII table with columns: Rank | Name | Email | Phone | Match% | Recommendation | Matched Skills | Missing Skills
   - Color-coded match %: green ≥80%, yellow 60-79%, red <60% (use chalk)
   - Truncate skills lists to 5 items with "...+N more"

2. formatJSON(result: SourcingResult): string
   - Pretty JSON with PII fields preserved
   - Add metadata: generated_at, jd_title, tool_version

3. formatMarkdown(result: SourcingResult): string
   - Full markdown report suitable for pasting into Notion/Confluence
   - Executive summary section, then candidate cards
   - Each card: name, contact, match%, fit_summary, skill match table

In /src/index.ts (CLI entrypoint):
- Accept args: --jd <filepath> --output <table|json|markdown> --top <number, default 10> --min-match <0-100, default 0>
- Read JD from file
- Run pipeline
- Print formatted output to stdout
- Write JSON result to ./output/results_{timestamp}.json always (audit trail)
- Show a progress spinner (use ora) during each pipeline phase with phase names
- On error: print actionable error message and exit code 1
```
